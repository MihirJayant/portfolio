Final Year Project37 TESTS · <500ms

Online Bidding System Git ↗

React · Node.js · Express · PostgreSQL · MongoDB · Socket.io · Gemini

Online auctions have a trust problem: bids land at the same instant, balances have to stay correct to the penny, and one accounting slip can let someone spend money they do not have. This full-stack platform solves that. People post listings and bid on them live, a Gemini-powered assistant guides them, and a virtual wallet keeps every balance provably correct. It runs a React front end over a Node and Express API, with PostgreSQL and MongoDB behind it, deployed to Render.

The core challenge was making the money trustworthy under concurrent bids, which I solved with an immutable double-entry ledger and ACID transactions, so no two bids can spend the same funds and every balance reconciles by design. Keeping bids in sync across users pushed me to learn Socket.io, and protecting personal data took me deep into JWT, OAuth, RBAC and the OWASP top ten. I taught myself the whole backend stack outside the curriculum, delivered it across eight Agile sprints, and proved it with a 37-test suite at 100% and sub-500ms responses. It is the project that taught me how to take a system from an idea all the way to production.

AI Chatbot70.38% ACC

WellAI Git ↗

Python · PyTorch · ResNet18 · Hugging Face · NLTK · OpenCV

WellAI is built for people who are struggling with their mental wellbeing and need a supportive, always-available space to talk. It holds a natural conversation, reads emotion from the user's facial expression, tracks sentiment as they type, and is designed so that someone in distress is met with a safe, appropriate response rather than a generic one. It is a multimodal system combining computer vision, classic NLP, and modern transformer models.

The hardest problems were accuracy and safety. I fine-tuned ResNet18 on the FER2013 dataset to 70.38% accuracy, then traced its mistakes through confusion matrices and found that fear and surprise were being confused because their facial cues look almost identical at 48x48 pixels. To keep responses safe, I built a regex guardrail that screens every message for crisis keywords before the model ever sees it, plus a TF-IDF retrieval engine with a carefully tuned similarity threshold so the bot stays on topic. Adding a Hugging Face model as an LLM-as-judge to score helpfulness, safety, and empathy taught me how to evaluate AI output rigorously instead of trusting a single accuracy number.

Distributed Systems4 ALGORITHMS

Load Balancer & Distributed File Transfer Git ↗

Java · JavaFX · Docker · MySQL · SQLite · AES

When a service gets busy, one server is not enough, and if a server fails the whole thing can go down with it. This project solves both: it spreads file uploads, downloads, and deletions across several machines and automatically routes traffic away from any node that becomes unhealthy, so the service stays fast and available under load. It is a distributed system in Java running across six Docker containers, with a live console showing node health and throughput.

The interesting challenges were all about concurrency and fairness. I implemented four scheduling algorithms (Round Robin, Weighted Round Robin, Shortest-Job-Next, and FCFS) and used semaphores with an aging mechanism to stop any request being starved under heavy load. Keeping data consistent across a local SQLite store and a remote MySQL database meant learning container-level locking and atomic operations, and building health-check failover taught me how real systems stay observable and recover from faults. It gave me a genuine feel for distributed architecture, Docker orchestration, and the trade-off between throughput and fairness.

Full-Stack · Cloud10,000+ RECORDS

CycleNest Git ↗

Java · Tomcat · REST · JavaScript · OSRM API · MongoDB · AWS EC2

CycleNest is a peer-to-peer marketplace that lets people rent bicycles and gear directly from one another. Its key feature is proximity: before booking, a user can see exactly how far an item is from them and how long it would take to reach. It is a full-stack project, a Java REST service built on Service-Oriented Architecture principles with a JavaScript front end that consumes the API and presents the results in the browser.

The central challenge was integrating a live external service. I consumed the OSRM routing API to calculate real distance and travel time between two locations, which meant writing robust error handling for timeouts, service unavailability, and malformed responses so the app never breaks when the third party does. Designing clean REST endpoints over the correct HTTP verbs and wiring the front end to them with fetch taught me how client and server actually communicate, and deploying to cloud NoSQL storage on AWS EC2, tested against more than 10,000 records, gave me hands-on experience with service-oriented, cloud-hosted design.

Big Data3-NODE CLUSTER

Distributed Big Data Analysis Git ↗

Java · Hadoop · HDFS · MapReduce · Linux

Some datasets are simply too large to analyse on one computer. This project takes the UK government's full road-safety record and processes it in parallel to find patterns in how and where serious road casualties happen, the kind of insight that can inform where safety effort should go. It runs on a three-node Hadoop cluster I built and configured from scratch on Linux.

Standing the cluster up was the real learning curve: configuring HDFS storage across the nodes, setting up inter-node communication, and running health checks before submitting any work taught me the operational side of distributed data that is usually hidden. Writing the analysis as Java MapReduce jobs meant thinking in terms of partitioning, shuffling, and aggregating at scale, and I made a point of validating every aggregate against known totals before trusting a result, the data-quality discipline real analytics depends on. It gave me a concrete understanding of how large-scale data platforms actually process information.

Robotics · CVNav2 + OpenCV

Autonomous Robot Navigation Git ↗

ROS2 · Nav2 · OpenCV · Gazebo · Docker · Python

This project tackles a core robotics problem: letting a robot move through an environment on its own, without bumping into things and while understanding what it sees. The robot detects obstacles and traffic signs, builds a map of the space around it, tracks its own position, and plans a safe route to where it needs to go. It is built on ROS2 and runs in a simulated world.

Reliable perception was the main challenge, using OpenCV to detect objects and recognise traffic signs in a real-time loop where a wrong reading immediately affects how the robot moves. Combining that perception with occupancy-grid mapping, visual odometry, and the Nav2 navigation stack taught me how the separate parts of an autonomous system have to work together, and running everything inside a Dockerised Gazebo and RViz setup gave me practical experience with both robotics tooling and reproducible environments. It is where computer vision stopped being theory and became something controlling a moving system.

Data Science5 MODELS

CRISP-DM Analytics Pipelines Git ↗

Python · pandas · NumPy · scikit-learn · matplotlib

Raw data on its own does not make decisions; it has to be cleaned, explored, and modelled before it tells you anything useful. This project takes real datasets through the full data-science lifecycle to turn them into predictions and clear insight, following CRISP-DM, the industry-standard process used in professional analytics. The emphasis is on finding which approach actually works, not just running one model.

I framed each problem, cleaned and explored the data, then compared five approaches (K-Nearest Neighbours, decision trees, logistic regression, clustering, and ensembles), judging them honestly through confusion matrices, ROC curves, cross-validation, and feature importance rather than a single score. Working through that comparison taught me why model evaluation matters as much as model building, and visualising the results for a non-technical reader pushed me to keep the focus on the decision rather than the maths. It gave me a repeatable, disciplined way to approach any analytics problem.
